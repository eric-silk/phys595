\section{Ratio-Consensus}
\subsection*{Ratio Consensus}
\begin{frame}
    \note[item]{Ratio consensus is an algorithm that enables coming to a
        consensus on the ratio of two values; in particular, when those ratios are
        ratios of sums as seen in this top equation.}
    \note[item]{If we model our communications between controllers as a directed
        graph, and then have each controller maintain two internal variables $y$ and
        $z$, we can update iteratively them according to these equations.}
    \note[item]{As this iteration progresses, it will asymptotically approach
        the true value $\gamma$.}
    \note[item]{So, as long as we can express our problem's governing equations
        in terms of ratios of sums, constants, and terms derivable from these, we can solve
        it in a distributed fashion.}
    \note[item]{As it happens, ratios of sums crop up \textit{everywhere} in
        engineering, and this is no exception!}
    \frametitle{Ratio Consensus}
    \begin{columns}
        \begin{column}{0.5\textwidth}
            Let:
            \[\gamma\coloneqq \frac{a_1+\ldots+a_n}{b_1+\ldots+b_n}=\frac{\sum_{i=1}^n a_i}{\sum_{i=1}^n b_i}\]
            Given a strongly-connected directed graph $\mathcal{G} = \{\mathcal{E},
                \mathcal{V}\}$, with each vertex $\mathcal{V}_j$
            posessing knowledge of their local contributions $a_i$ and $b_i$ and
            maintaining a pair of internal states $y_j$ and $z_j$, as well as their
            in-neighbors and out-neighbors.

            \textbf{Note}: a \textit{directed} graph implies one-way communications are acceptable!
        \end{column}
        \begin{column}{0.5\textwidth}
            \begin{figure}
                \includegraphics[width=\textwidth]{img/graph.png}
            \end{figure}

        \end{column}
    \end{columns}
\end{frame}

\begin{frame}
    We can iteratively update these states as
    \[ y_j[k+1]=\sum_{i\in\mathcal{N}_j^-}\frac{1}{D_i^+}y_i[k]\]
    \[ z_j[k+1]=\sum_{i\in\mathcal{N}_j^-}\frac{1}{D_i^+}z_i[k] \]
    where $\mathcal{N}_j^-$ is the in-neighborhood of $j$ and $D^+_i$ is the degree of the out-neighborhood of $i$.
    Then, take their ratio, which will converge to $\gamma$:
    \[\gamma_j[k] = \frac{y_j[k]}{z_j[k]},\
        \lim_{k\rightarrow\infty}\gamma_j[k]\rightarrow \gamma\forall j\]
    \note[item]{Point out it's assumed (although probably not required) that every node has a self-loop.}

\end{frame}

\begin{frame}
    \note[item]{The described ratio consensus algorithm assumes some level of
        synchrony and lossless transmissions. If we expect packet loss, how can we
        robustify our algorithm?}
    \note[item]{Transmitting running sums works to get around this issue. Each
        node transmits its historical values of $y$ and $z$ (stored as a sum, no
        need to send the whole array!)}
    \note[item]{Then, the update for $y$ and $z$ is simply taking the difference
        between the latest received value and the previously stored one.}
    \frametitle{Robust Ratio Consensus}
    \textbf{Q:} If not using a reliable transport method (e.g. TCP/IP), how do we handle
    packet drops?\newline
    \textbf{A:} We transmit \textit{running sums}! \cite{robust_rc}\newline
    Let:
    \[
        \sigma_{j}[k+1] = \sum_{t=0}^{k}\frac{y_j[t]}{\mathcal{D}^+_j},\
        \beta{j}[k+1] = \sum_{t=0}^{k}\frac{z_j[t]}{\mathcal{D}^+_j}
    \]
    Each node $j$ locally tracks the $\sigma$'s and $\beta$'s received from its in neighbors, $i\in\mathcal{N}_j^-$,
    notated as $\theta_{ji}$ and $\phi_{ji}$. Then:
    \[
        y_j[k+1] = \sum_{i\in\mathcal{N}_j^-}\left(\theta_{ji}[k+1]-\theta_{ji}[k]\right),
        z_j[k+1] = \sum_{i\in\mathcal{N}_j^-}\left(\phi_{ji}[k+1]-\phi_{ji}[k]\right)
    \]
    \note[item]{While the sums won't converge, their \textit{ratio} will!}
    \note[item]{Convergence of RC is \textit{exponential}, whereas the sums grow \textit{linearly}}

\end{frame}

\subsection{Applications}
\begin{frame}
    \frametitle{Applications}
    \begin{itemize}
        \item Secondary Voltage Control
        \item Secondary Frequency Control
        \item Optimal Power Flow
        \item Economic Dispatch
        \item Any other algorithm that can be formulated in terms of ratios of sums
    \end{itemize}
\end{frame}

\begin{frame}
    \frametitle{Secondary Frequency Control}
    For a given grid, e want to maintain some system frequency $f_s = f_{nom}$ (50
    or 60Hz). Load-generation mismatch may lead to $f_s \neq f_{nom}$. Furthermore,
    individual buses may see local variations, i.e.:
    \[ f_0 \neq f_1 \neq \ldots \neq f_n \neq f_{nom} \]

    To determine a generation set-point for secondary frequency control, we need one
    $f_{s}$ from all these local values.
\end{frame}
\begin{frame}
    \frametitle{Secondary Frequency Control}
    Let:
    \[f_s \coloneqq \frac{\sum_{i=0}^{n} f_i}{n}\]
    That is, $f_s$ will be the \textit{average} of the local frequencies. This can be formulated via RC:
    \[ f_s = \frac{f_0 + f_1 + \ldots + f_n}{1 + 1 + \ldots + 1} \]
    which can then be fed into your secondary control algorithm of choice (droop, etc.)
\end{frame}

\section{Robust...but Trusting}
\begin{frame}
    \frametitle{Robust...But Trusting}
    We've seen how we can use this algorithm, and how it can be robustified to packet loss.
    But given this is a security course:

    \begin{center}
        \textbf{How do we handle malicious, compromised, or faulty nodes?}
    \end{center}
\end{frame}

\begin{frame}
    \frametitle{Disclaimer}
    Due to a failure in communication (the irony is not lost on me), the work
    I'm doing is based on unpublished work of my advisor. He's requested I not
    share details, so we're instead going to take a 10k foot view of
    establishing trust in this algorithm.
\end{frame}

\begin{frame}
    \frametitle{Trust, but verify}
    A fundamental method for establishing trust in the security management of
    people is the use of multiple third parties.
    \begin{itemize}
        \item Identify associates of the individual in question
        \item Solicit information from them about
              \begin{itemize}
                  \item The individual
                  \item Other contacts
              \end{itemize}
    \end{itemize}
    Even if the individual is able to collude with one associate, multiple associates becomes harder.
    Colluding with associates of associates quickly becomes prohibitive.
    \newline
    \begin{center}
        \textbf{Our graph model should be extended to mimic this practice}
    \end{center}
\end{frame}
\begin{frame}
    \frametitle{Algorithms Ain't People\footnote{With apologies to our future AI overlords}}
    Unlike people, algorithms are built on axioms and can have truly provable
    behaviors (within the reality imposed by the axioms). Furthermore, it can exhibit
    \textit{invariants}, properties that remain unchanged under various operations and transformations.

    \begin{center}
        \textbf{Our algorithm should be examined for provable invariants}
    \end{center}
\end{frame}

\begin{frame}
    \frametitle{As luck would have it...}
    ...both of these details pan out for this underlying algorithm. In short, a node can:
    \begin{itemize}
        \item Track some invariant value
        \item Advertise this to our out-neighbors' out-neighbors
        \item Use the advertised value from our in-neighbors' in-neighbors and
              compare it to our local value
        \item Use \censor{some fake text for humor} and then \censor{stand there awkwardly}
        \item \censor{laugh please I am begging you} because \censor{I'm slapping this all together at the last minute}
    \end{itemize}
\end{frame}
\begin{frame}
    \frametitle{Handling a loss of trust}
    Once a node has been declared untrustworthy via the prior algorithm, we can
    remove it from our in-neighborhood. Several futher steps could be taken:
\end{frame}
\begin{frame}
    \frametitle{Removal of Influence}
    A naive implementation might just throw away the entirety of our work so
    far. However, with some cleverness, we could remove only the contributions
    of the faulty node and continue the current run
\end{frame}
\begin{frame}
    \frametitle{Removal of Node}
    Of course, we probably won't want to trust this node for future runs.
    \begin{center}
        \textbf{We can choose to remove this node from our in-neighborhood and never hear from it
            again.}
    \end{center}
\end{frame}
\begin{frame}
    \frametitle{Ostracize}
    \begin{center}
        \textbf{We can advertise this loss of trust to our out-neighbors so they can take the appropriate steps.}
    \end{center}
\end{frame}
\begin{frame}
    \frametitle{Banish them to the Shadow Realm}
    Of course, alerting everyone to this loss of trust is a double-edged sword. While it may help
    prevent failures on other nodes, it may also alert the node in question.
    \begin{center}
        \textbf{We can use multiple unicast messages to selectively inform out-neighbors that aren't the faulty node is it is unaware of its loss of trust.}
        \linebreak\linebreak\noindent
        This mimcs the social practice of \textit{shadow banning}.
    \end{center}
\end{frame}

    \end{itemize}

\begin{frame}
    \frametitle{CMake}
    A build generator -- it builds the instructions for your preferred build tool\footnote{Abstracts away specifics of various platforms}
    \begin{columns}
        \begin{column}{0.5\textwidth}
            \begin{itemize}
                \item Define compilation variables
                \item Automagically bring in dependencies
                \item Define compilation targets
                \item Toolchains: support cross-compilation or non-standard compilers
                \item Presets: convenient, named configurations for repeatable builds
            \end{itemize}
        \end{column}
        \begin{column}{0.5\textwidth}
            \begin{figure}
                \includegraphics[width=0.45\textwidth]{img/cmake_screenshot.png}
            \end{figure}
        \end{column}
    \end{columns}
\end{frame}
\begin{frame}
    \frametitle{Unit Tests: GoogleTest, GoogleMock}
    \begin{itemize}
        \item Enables detailed, specific scenarios to verify correct operation of individual components (``units'')
        \item Allows you to pinpoint cause of failures before integration/deployment
        \item Use of ``mock'' objects enables testing of one unit without dependence on another unit
        \item Also using the framework for ``smoke'' and basic ``integration'' tests
        \item Currently over 120 tests written, more to be done
        \item Will be verifying coverage using gcov
    \end{itemize}

\end{frame}

    \begin{itemize}
        \item What attacks can be perpetrated against this method?
        \item For these attacks, how can we:
              \begin{itemize}
                  \item Identify
                  \item Mitigate
                  \item Prevent
              \end{itemize}
        \item ``Shadow-banning'' faulty nodes (pending paper by ADG, stay tuned...)
        \item Homomorphic encryption (leverage templating/operator overloading!)
        \item What sort of attacks might be enabled \textit{with} this method?

                  \item I.E. if we've compromised devices and have constrained communication channels, can this be used to coordinate other attacks?
              \end{itemize}
